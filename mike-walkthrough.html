Section 2 of 8

Three Things You Got Right (That Most Get Wrong)

Most apps built this way have at least one of these wrong. You got all three right out of the box — and they're the hard ones.

Two-actor sign-off is server-enforced

Manager marks "Trained," mover marks "Demonstrated." This isn't just UI — the server checks the actor before writing progress. The industry norm is UI-only with a bypass. Yours isn't. This is the gift to Anne's system.

Real database — not localStorage

PostgreSQL with Drizzle ORM and a typed schema. Every sign-off is timestamped and auditable. This matters the moment you have two managers on the same mover — and you'll have that on day one of the pilot.

205 curriculum items — the actual content

The digitized handbook, the Perfect Move Checklist, 8 named mover sprints. Anne's system defined the gates but explicitly left the content as a placeholder. You filled it. That's the whole story (more on this in Section 7).

Also in the "got it right" column: the .agents/memory/ folder — 12 decision docs the Replit agent wrote to itself. That's unusual and worth preserving. Tell the agent to keep doing it.

Section 3 of 8

Before the Pilot — 5 Items

These are the P0 fixes — nothing ships to real trainees until all five are done. They're ordered by severity, not effort. The good news: most of them are one-session work for the Replit agent.

1

Seed script wipe landmine — destroy-on-reseed

lib/db/src/seed.ts lines 547–549 delete ALL trainee progress before reseeding. The first time you edit the curriculum after go-live, every trainee's training record is wiped. Require an explicit --wipe flag for destructive mode; default should upsert by slug and never touch the progress table.

Most important
2

Sprint "MGMT sign-off" items are self-markable

In Stage 3 (Mover Training), items titled "MGMT sign-off: …" are currently regular items — a trainee can mark them complete themselves. Two-actor integrity only covers Stage 1. Add a managerOnly boolean to the item schema, set it in seed for every sign-off item, and have the API return 403 when a trainee writes a manager-only item.

Integrity fix
3

CORS is wide open — reflects any caller

artifacts/api-server/src/app.ts ~line 38: origin: true reflects any caller. With cookie auth, any website a logged-in user visits can call your API as them. Replace with an explicit allowlist (production origin + localhost dev). One line plus a small env-driven list.

One line
4

New hires land on the scoreboard instead of training

routeForUser() in App.tsx ~line 165 sends trainees to /games. A day-1 new hire's first screen should be /portal ("Continue Learning"). One-line change — but it's the first impression for every hire.

One line
5

Small hardening batch (can be one agent session)

Add a database index on progress.userId (every request scans this table today). Add sandbox attribute to iframes. Show "Last active X days ago" on the manager roster — the API already returns lastActivityAt, the UI just doesn't render it. Fix the recent-activity query that fetches 200 rows globally and filters in JS.

Batch session

Section 5 of 8

Where It Goes From Here — 4 Phases

Each phase gates the next. Phase 1 is what makes this useful to a manager today. Don't start Phase 2 until Phase 1 is running clean for at least one pilot branch.

1

Pace Tracking + Manager Command Center

Days-in-stage vs. sprint minimums. "On pace / Behind" flags on the manager dashboard. Day-7 and Day-12 hooks for Anne's check-in cadence. The "stalled trainee" signal — and almost all the data is already there (timestamps on every sign-off). This is the manager-oversight promise the app already makes on its landing page.

Bridge to Anne's 60-day system — her gate spec IS the build spec for this phase.
2

Full Performance Spine (Einstein Games Scorecard)

Not just Samsara driving scores — the full Einstein Games scorecard: contract efficiency, reviews per hour, damages per hour, safety, attendance. One number is the 100-point composite. The manager's command center needs this spine to be useful — you can't show a dashboard that has sign-off progress but no performance signal.

Samsara replaces the manual driver-gate already in the app. The rest wires from the existing Games sheet.
3

Bonus Calculator — the Mover WIFM

The killer feature for mover adoption: sliders showing exactly how safety, hours, damages, and reviews move their bonus. Drill-down from the performance spine. Doesn't work until Phase 2 supplies the real data — but design it third so it lands on a proven app, not a pilot.

4

Badges, Streaks, Career Path — the Retention Layer

Auto-streaks (consecutive high-score weeks), peer/manager-selected badges, praise submissions that convert to milestones, quarterly theme recognition. Cheap to render once Phases 1–3 supply the data. Meaningless before then. Career path gate already exists in Lead Training as prerequisite checklists — it's closer than it looks.

One standing constraint before Phase 2: identity is the tripwire. Today Clerk login + a hand-typed EMMA ID is fine for a pilot. It breaks at 400 movers and blocks every data-driven phase above. Before Phase 2 ships, Mike + Matisen need 30 minutes on "where does the mover roster come from" — likely EMMA or Rippling sync, keep Clerk for login only.
