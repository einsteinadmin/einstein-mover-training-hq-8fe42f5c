6/24/2026
Infra Feature Content

Test infrastructure hardened, praise workflow built, photo sign-offs, video gates expanded — major build session

Data protection & test infrastructure

  • Seed/push-force regression tests — data-protection guardrails now have regression tests for the "refuse to wipe" latches on seed and push-force commands.
  • Curriculum migration regression test — regression test added that runs a migration against a seeded throwaway database and asserts trainee progress survives and is re-attached to the correct items after restructure. Closes the gap where a future migrate-*.ts script could silently drop or re-key progress rows on a populated/prod DB.
  • Lead Training migration coverage — equivalent safety net added for the Mover Training (lead-in) curriculum migration. When Lead Training applies the same sprint + review restructure, the new migration has automated coverage from the start.
  • Ephemeral DB test harness (describeDb + globalSetup) — DB-backed safety suites now run reliably via an ephemeral Postgres globalSetup (Task #27). Tests no longer silently skip in environments without a provisioned test database, eliminating the false-green problem.
  • Additive-schema assertion — replit.md's promise that schema changes are always additive (add columns/tables, never drop) is now automatically enforced. A future edit that drops or renames a column will turn the build red.
  • Deploy gate wired — DB-backed test suite now runs as part of the production build (build:production in api-server/package.json, wired via artifact.toml). A regression that would wipe trainee progress or break a curriculum migration is caught before it ships. Note: deploy build environment must have initdb/pg_ctl/psql on PATH for DB suites to run; a missing-tooling situation is designed to fail loudly rather than silently skip — environment compatibility should be confirmed before next publish.

Training content & curriculum

  • Booklet line items reworked — initial booklet upload had broken all text lines into separate completion items. Went through the full booklet and reworked all line items to match the original structure. Steps that belong together are now grouped as checklists within items rather than separate sign-offs.
  • Video gates expanded — "Watch:" video gate steps added to additional Mover Training and Lead Training sprints, matching the Sprint 2 (Home Prep) pattern. Each video step sits at the top of its sprint and must be watched to completion before hands-on tasks unlock. Placeholder "coming soon" clip used for all new gates; real footage swaps in via each item's videoUrl.

Praise workflow & photo sign-offs

  • Structured praise check-ins — praise check-ins now require the trainee to write actual praise using a structured fill-in-the-blank format. Both narrative fields must contain a real sentence before submission is allowed. Submission routes to the manager dashboard for explicit approve/deny before the step counts as done.
  • Photo attachment + manager approval on sign-offs — select hands-on tasks (e.g., furniture wrapping) now require the trainee to attach a photo at each sign-off. Managers review photos on the dashboard and can deny a single sign-off with written feedback — removing only that attempt, not other completions for the same item — and the trainee sees the feedback and can redo it.
6/22/2026
Feature Integrity Infra

Sprint review workflow, video prerequisites, object storage, and data guardrails — major build session

Nine updates — two categorized as big lifts, two critical infrastructure investments:

Sprint structure & gating

  • Instructions gate on all sprints — all sprints locked until the instructions and expectations doc is marked read.
  • Sprint heading cleanup — heading structure cleaned up across both Mover and Lead training modules.
  • Multi-completion sprint items — sprint items in Mover and Lead modules now require 2+ "Completed" confirmations to mirror the existing handbook's proficiency model. Second completions are date-locked: only one completion per item per day, preventing a trainee from marking the same item done twice in the same session.
  • Sprint 1 reworked — sprint 1 restructured to better align with intended coursework.

Between-sprints review workflow BIG LIFT

  • Mover Training stage restructured — each sprint now contains only hands-on skill items. Cross-cutting "sprint completion" items (praise, culture exercises, eNPS surveys) moved out into a new between-sprints review step.
  • Manager sign-off gates next sprint — when a trainee finishes the review step, the sprint is flagged as "Ready for Review" on the manager dashboard. Manager sign-off on the review step is what unlocks the next sprint. The hand-off to management is now an explicit, visible checkpoint instead of a self-marked line buried inside the sprint.

Video prerequisites BIG LIFT

  • Video hard gate on select sprints — specific Mover Training sprints now require the mover to watch a training video to completion before any item in that sprint can be marked complete. Same model as the existing "Start Here" reading gate. Unlocks the sprint's hands-on work only after the video is finished.

Infrastructure & data integrity

  • Object storage added — app now uses object storage for large-format content (docs, videos) to avoid bloating the static app and risking slowdowns or crashes. One-time investment: all videos downloaded and re-uploaded into the new storage layer.
  • Backend guardrails + server tests — multiple backend guardrails and server-side tests added to prevent process regression. Patches several edge cases surfaced during development review.
  • Trainee progress permanently protected — app now has hard guardrails on all destructive commands and a safe-publish process. Future deployments and schema changes cannot wipe trainee progress, sign-offs, or manager dashboards.

  • "What we're looking for" reformatted — all eval point descriptions converted from paragraph to bullet form for easier reading in the field.

Critical Lock CORS to known origins. Current config reflects any caller — combined with credentialed requests, this lets any attacker site call the API as a logged-in manager. One-line allowlist fix in api-server/src/app.ts.
Critical Guard the seed script from destroying live progress. Re-running seed.ts currently deletes ALL trainee progress with no warning. Add a --wipe flag for destructive mode; default = upsert curriculum, never touch progress rows. Most important fix in the package. ✓ Resolved 6/22: Hard guardrails added to all destructive commands + safe-publish process in place. Build wing

The module roadmap

Cameron's ranking as of 6/12. Each module builds on the one before it — data readiness, not coolness, drives the order.

#1 — Next to build

Pace tracking + manager command center

Days-in-stage vs sprint minimums, on-pace/behind flags on the manager dashboard, Day-7 and Day-12 hooks. This is the bridge to Anne's 60-day performance system — her gates are literally the build spec for this module. Almost entirely built from data the app already has (timestamps exist on every sign-off; only stageStartedAt is new).

Cameron's extension: managers can eventually schedule training sessions and check-ins forward-looking — not just record what happened.

#2

The performance spine

Surface the whole Einstein Games scorecard natively: contract efficiency, reviews/hr, damages/hr, safety scores, attendance (call-outs and late arrivals). Samsara is one input here — not the headline. The Performance Log page already reads the Games sheet; this module adds trends, rings, and coaching tips from the mockup. Much of the data already flows into the app.

Note: Samsara scores already have a driver-test gate in the app as manual entry — this module replaces the manual logging with live data.

#3

Bonus calculator

The biggest mover-WIFM in the whole prototype: sliders showing how safety scores, hours, damages, and reviews actually move your bonus number. Heaviest data dependency — needs payroll-grade numbers and the full comp formula. Do it third so it lands on a proven app with real users, not a pilot.

#4

Badges + streaks + the quarterly scoreboard

The reward and motivation skin that rides on the performance spine. Multi-source: auto streak data from the spine + peer/manager-selected badges + praise submissions from Monday. Long-term, the app carries the live quarterly-theme scoreboard here too. Cheap to render once modules 1–3 supply the data; meaningless before then.

Identity tripwire — this is the gating dependency for modules #2 and #3.
Today: Clerk login + hand-typed EMMA ID per user. Fine for a pilot; breaks at 400 movers and blocks every data-driven module (Samsara, bonus, attendance all need to know which mover this is). Before module #2 ships, Mike + Matisen need 30 minutes on "where does the mover roster come from" — likely answer: sync from EMMA or Rippling, keep Clerk for login only. The EMMA API/MCP (Matisen's track) is the unlock.
